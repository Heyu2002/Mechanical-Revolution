# 任务识别的前沿方案研究

## 当前主流方案的局限性

我之前提供的方案（规则匹配、LLM 分类、评分系统）确实是比较传统的做法，存在以下问题：

1. **过于依赖预定义规则** - 难以适应新场景
2. **二元判断思维** - "是任务" vs "不是任务" 过于简单
3. **忽略了用户意图的连续性** - 任务可能是渐进式的
4. **缺乏学习能力** - 无法从历史交互中改进

## 更先进的思路

让我重新思考这个问题的本质：

### 问题的本质

**任务的分界点不应该是一个判断，而是一个状态转换**

用户的输入可能处于多种状态：
- 探索阶段（"我想做点什么..."）
- 需求澄清阶段（"具体来说..."）
- 任务确认阶段（"帮我实现..."）
- 任务执行阶段（实际工作）
- 任务调整阶段（"改成..."）

### 几个关键洞察

1. **任务不是离散的，而是连续的**
   - 不是"是/否"，而是"任务化程度"（0-100%）

2. **任务的边界是模糊的**
   - "什么是 React？" → 可能只是好奇
   - "什么是 React？我想学习" → 可能需要学习计划
   - "什么是 React？帮我写个教程" → 明确任务

3. **用户可能不知道自己要什么**
   - 需要引导式对话
   - 逐步明确需求

## 我想听听你的想法

在你看来，什么样的方案才是"符合预期"的？

是否是以下方向：

### 方向 A：Agent-as-a-Service
- 每个输入都当作潜在任务
- 由 agent 自己决定如何响应
- 没有预先的"任务判断"层

### 方向 B：意图理解 + 主动澄清
- 不急于判断是否是任务
- 先理解用户意图
- 如果不明确，主动询问
- 逐步构建任务定义

### 方向 C：任务状态机
- 维护一个任务状态
- 每个输入都可能改变状态
- 状态决定系统行为

### 方向 D：基于 Prompt 的自适应
- 在 Orchestrator 的 prompt 中定义任务判断逻辑
- 让 LLM 自己决定是否需要分解
- 系统只提供工具，不做预判断
